## ■ 目的

このマップは以下のために存在する：

1. **観測者の B（目的・用途）に応じた読み順を提供する**
2. **文書間の依存・参照関係（W_ij ラベル M_B ネットワーク）を可視化する**
3. **バージョン同期・整合性チェックの基準を提供する**
4. **新規文書を追加する際の位置決め指針を与える**

このマップは **README とは独立** に運用される。README が「初見者向けの読み方ガイド」であるのに対し、このマップは「文書間関係の詳細マップ」である。

※ 注意：当プロジェクトは現在、名称を「RDF（Framework）」から「RDL（Language）」へと移行中です。文書名や内部記述にはまだ「RDF」が多く残っていますが、これらは実質的に「RDL」を指すものとしてお読みください。

## ■ 分類スキーマ

### 軸1：Tier（依存階層 / OS的役割分担）

| Tier | 名称 | OS対応 | 役割 |
|---|---|---|---|
| **T0** | 哲学的基底層 | Foundation | 前提となる世界観 |
| **T1** | Core層 | **Kernel** | 最小ダイナミクスと基底量 |
| **T2** | 運用層 | Runtime / Shell | Core を動かすための操作論 |
| **T3** | 外部接続層 | Driver / Adapter | 他理論を Core 接続用に取り込んだ層 |
| **T4** | 応用層 | Application | 特定領域への適用 |
| **T5** | メタ層 | Reflection / Introspection | 自己食的・自己参照層 |
| **T-D** | 共有語彙・対応層 | Shared Library / Vocabulary Layer / Headers | 全層から参照される語彙基盤。RDL用語⇄自然言語の双方向変換を担う |

> **OS 対応の注記**
>
> - Tier 区分は OS アーキテクチャの役割分担と構造的に類似する。説明補助として有用。
> - **同一性主張ではない**。OS は決定論前提が強く、ξ 残存性のような RDL 固有概念は OS 語彙では完全には表現できない。
> - 他のメタファー（生命体・言語系・ネットワーク）も B 依存の別カットとして並立する。
> - OS ラベルは外部協力者・工学者への説明時に説明コストを下げるための杖として運用する（道具特性 v0.1 と整合）。

### T-D：共有語彙・対応層（Shared Library / Vocabulary Layer / Headers）

## ■ このマップの自己食的注意書き

このマップ自体も SILN（M_B）であり、以下のリスクを持つ：

1. **分類の固定化**：Tier やカテゴリが「絶対的な真理」として扱われるリスク
2. **網羅性の錯覚**：このマップに載っていない文書が「存在しない」と扱われるリスク
3. **依存グラフの単純化**：実際の文書間関係は、ここに描かれたツリー構造より複雑
4. **同期コスト**：このマップ自体の更新が滞ると、本体文書との乖離が H_vec として蓄積
5. **メタファーの道具化（v0.2 新設）**：OS 対応注記が「真の構造」として固定化されるリスク。OS は便利な杖の一つに過ぎず、生命体・言語系・ネットワークなど他のメタファーも並立する

### 破断条件

- Tier に収まりきらない文書が増えた場合
- カテゴリ横断が常態化した場合
- Status の遷移が頻繁になり追跡コストが上がった場合
- このマップを更新せず本体だけ更新する状態が続いた場合
- **OS メタファーが説明補助ではなく定義的意味を持ち始めた場合（v0.2 新設）**

### 残存構造（このマップが破綻しても残るもの）

- 「Core を中心としたハブ＆スポーク構造」という設計思想
- 「観測者の B 別に読み順を提供する」というアプローチ
- 「自己食的注意書きを伴う運用」という姿勢
- **「複数メタファーを並立させる」というラベル運用観（v0.2 新設）**
