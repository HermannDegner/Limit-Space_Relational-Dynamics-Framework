# RDL_切替判定 v1.0
*T2：運用層 / OP / ACTIVE*
*依存：RDL_Core / 借用モジュール群（NAM / GR-MOD / FluctMOD）*

---

## ■ 0. 位置づけ

本文書は、RDL 系を実際に運用するとき、**どのモジュールを使うかを決める判定論**である。

切替判定は Core にも借用モジュールにも置かない。
Core に置くと Core が自己選択することになり循環参照が生じる。
借用モジュールに置くとモジュールが自分自身を推薦することになり判定が壊れる。
したがって切替判定は独立した運用層文書として存在する。

### 本文書が扱うこと

```text
・目的から入る判定の手順
・対象 SILN の「動かすか／背景に置くか」の判定
・解像度と計算負荷のトレードオフ操作
・トレードオフをどう操作するかの手順論
・モジュール選択の補助表
・追従運用（一回判定で終わらない）
```

### 本文書が扱わないこと

```text
・各借用モジュールの内部数理     → 各借用モジュール文書を参照
・SILN 分解の操作手順           → SILN分解の数理フレームワーク v2.2 を参照
・モジュールの耐久検査           → 構造破壊学 前提文書 v0.4 を参照
・運用上の道具化リスク           → RDL道具特性 v0.1 を参照
```

## ■ 1. 切替判定の基本構造

切替判定はパラメータ表から始まらない。

**目的から始まる。**

「D[ξ] < ε なら NAM」という条件は正しい。
しかし実際の運用で最初に手元にあるのはパラメータ値ではなく、
「何のためにこれを動かすのか」という問いである。

目的が決まると、扱うべき SILN が見える。
SILN が見えると、必要な解像度が見える。
解像度が見えると、許容できる計算負荷が見える。
ここで初めてモジュール選択が意味を持つ。

```
目的の確認（B の確認）
  ↓
対象 SILN の洗い出し
  ↓
各 SILN を「動かすか／背景に置くか」を判定
  ↓
動かす SILN に「どの解像度か」を決める
  ↓
解像度 × 許容負荷 → モジュール選択
  ↓
運用しながら引き直す（追従運用）
```

---

## ■ 2. ステップ 1：目的の確認（B の確認）

最初に問うのは、**何のためにこれを動かすのか**である。

B を確認することは、以下の四つを決めることである：

```text
① 何を見たいのか（評価関数）
② どのスケールで見たいのか（空間スケール・時間スケール）
③ どの程度の精度が必要か（解像度要求）
④ 誰が使うのか・いつ使うのか（用途・観測者）
```

これらは独立ではない。評価関数がスケールを決め、スケールが解像度要求を決め、解像度要求が許容負荷の上限を決める。

### 目的の典型例

| 目的 | 評価関数の方向 | スケール感 |
|---|---|---|
| 生態系のNPC的動態を動かしたい | 個体行動の多様性 | 個体〜集団 |
| 社会制度の熱蓄積を見たい | 整合崩壊の兆候 | 集団〜文明 |
| 個体の性格動態を記述したい | 動態の有用な近似 | 個体 |
| 物理スケールの構造変化を扱いたい | 幾何・スケール支配 | 天体〜宇宙 |
| 創作世界の整合性を保ちたい | 物語の内的一貫性 | 世界設定依存 |

目的が曖昧なまま次のステップに進むと、解像度の判定が揺れる。
まず B を仮設する。あとで引き直せばよい。

---

## ■ 3. ステップ 2：対象 SILN の洗い出し

目的が決まったら、関係する SILN を並べる。

重要なのは、**「扱える」と「必要か」は別の問いである**、という点だ。

RDL は原理的に地形から個体の感情まで動的構造として扱える。
しかし、それをすべて動かす必要があるかは、目的次第である。

生態系シミュレーターを例にとると：

```text
扱える SILN 候補：
  地形・気候・植生・水系・個体NPC・群れ・天候・地殻変動 …

目的が「個体NPCの行動多様性を動かす」なら：
  → 地形：背景 SILN（変化しない前提で十分）
  → 天候：中間 SILN（NPCに影響するなら軽く動かす）
  → 個体NPC：動かす SILN（中心）
  → 地殻変動：不要（タイムスケールが目的と合わない）
```

### 分類の判定基準

```text
動かす SILN   ：目的の評価関数に直接影響する
中間 SILN     ：目的に間接影響する。軽量近似で十分な場合が多い
背景 SILN     ：目的の時間・空間スケールでは変化しない
不要 SILN     ：目的のスケールと合わない・影響が観測不能
```

この分類は仮設である。運用中に「背景に置いたが影響が出た」と分かれば引き直す。

---

## ■ 4. ステップ 3：解像度の決定

動かす SILN が決まったら、**どの粒度で動かすか**を決める。

解像度には二段階ある（AI仮想世界シミュレータ v0.3 §3 に対応）：

```text
SILN 段階（低解像度）：
  SILN を塊として動かす。内部構造は開けない。
  「天候 = 晴れ/雨/嵐」程度の粒度。
  計算負荷が低い。整合性管理が単純。

M_B 段階（高解像度）：
  SILN に境界 B を引いて内部の M_B ノードまで開ける。
  「天候 = 温度・湿度・気流・降水量の相互作用」まで扱う。
  計算負荷が上がる。整合性管理が複雑になる。
```

**解像度は目的が要求する最低水準に合わせる。それ以上は過剰である。**

過剰な解像度は：
- 計算負荷を上げる
- 管理すべき M_B ノードが増える
- 整合性破綻のリスクが増す
- 目的に関係しない熱蓄積点が増える

「できる」と「必要」のせめぎあいが、解像度決定の核心である。

## ■ 5. ステップ 4：計算資源制約と理想解像度のせめぎあい

ここが切替判定の実質的な中心である。

### せめぎあいの構造

```text
理想の解像度（評価関数が要求するもの）
  ↕  ← ここがせめぎあい
計算資源制約（変えられない B）
```

理想と制約は同時に満たせないことが多い。
このせめぎあいは、一回の判断で解決するのではなく、**何を落とすか**を決める操作として扱う。

### 「落とせるか」の判断基準

```text
落としても目的に影響しない  → 背景 SILN 化・SILN段階に落とす
落とすと目的が壊れる        → 別の近似を探す・負荷の低いモジュールを使う
落とし方が分からない        → まず最低解像度で動かして観測する
```

### 操作の流れ

```text
1. 理想の解像度で設計する
     ↓
2. 計算資源制約にぶつかる
     ↓
3. 「どこを落とせるか」を判断する
     ↓
4. 落とした部分が目的に影響するか確認する
     ↓
   影響しない → 背景に落とす・軽量近似に切り替える
   影響する   → 別の近似を探す（モジュール変更・SILN段階化）
     ↓
5. 運用しながら引き直す
```

この操作は一回では終わらない。運用中に制約も目的も変化する。
切替判定は追従運用である。

## ■ 6. ステップ 5：モジュール選択

解像度と許容負荷が決まると、モジュールが選べる。

### 選択の基本方針

```text
できるだけ軽いモジュールを使う。
軽いモジュールで目的が果たせないときだけ重いモジュールへ上げる。
重いモジュールを使うのは「必要だから」であって「できるから」ではない。
```

### モジュール選択補助表

| 条件 | 推奨モジュール | 理由 |
|---|---|---|
| ξ が目的に影響しない・無視できるスケール | **NAM** | 最軽量。確定近似。D[ξ]≈0, H<<θ_eff が条件 |
| ξ が本質的に効いている・揺らぎが目的の核 | **FluctMOD / Core** | 揺らぎを落とすと目的が壊れる |
| 大スケール・幾何構造が支配している | **GR-MOD** | 時空曲率・大質量スケール |
| 揺らぎの統計分布そのものを扱いたい | **FluctMOD** | 分布・確率・統計借用 |
| 上記のいずれでもない・汎用 | **Core** | 最小ダイナミクス。借用なし |
| 複数条件が混在する | **ハイブリッド** | 部分ごとに別モジュール |

### モジュールの負荷感

```text
軽 ──────────────────────────── 重
NAM ← FluctMOD ← Core ← GR-MOD
```

NAM は最軽量だが、ξ を落とした近似であることを忘れない。
GR-MOD は大スケール向けで、小スケールに使うと過剰になる。

### ハイブリッド運用

一つのシミュレーション内で異なるモジュールを混在させることができる。

```text
例：生態系シミュレーター

  地形（背景 SILN）   → 扱わない
  天候（中間 SILN）   → NAM（ξ無視の軽量近似）
  個体NPC（動かす）   → Core + FluctMOD（揺らぎが個体差の源）
  群れ動態            → NAM（集団レベルでは揺らぎが平均化）
```

部分ごとにモジュールを変えることは、複雑さを増すが整合性を保つことができる。
ハイブリッドにするなら、境界 B の接合点（どこでモジュールが切り替わるか）を明示する。

## ■ 7. 追従運用：切替判定は一回で終わらない

切替判定は静的な決定ではない。

```text
運用 → 観測 → 「落としたところに影響が出た」
                  ↓
             引き直し（解像度を上げる・モジュールを変える）

運用 → 観測 → 「ここは思ったより重要でなかった」
                  ↓
             引き直し（背景に落とす・軽量化する）
```

切替判定の文書は「正解の判定表」ではない。
**目的と制約のせめぎあいを操作する手順論**である。

---

## ■ 8. 切替判定の破断条件

切替判定自体が機能しなくなる条件：

```text
・目的が未確定のまま判定しようとする
    → B が決まらないとモジュール選択が揺れる

・「できるから使う」でモジュールを選ぶ
    → 過剰解像度による負荷増大・整合性破綻

・一回の判定で終わりにする
    → 追従運用しないと目的変化・制約変化に対応できない

・モジュール間の境界 B を明示しないハイブリッド運用
    → 接合点が不明だと整合性が保てない

・パラメータ条件表だけに頼る
    → D[ξ] を計測していない段階では表が使えない
```

## ■ 9. 構造耐久証明書

### 基本スペック

```text
ラベル        ：RDL_切替判定 v1.0
物理構造      ：RDL 系の運用者・設計者
中核構造      ：目的から入るモジュール選択の手順論
上層構造      ：RDL_Core / 借用モジュール群 / 道具特性 v0.1
```

### 境界仮定

```text
観測者        ：RDL を実装・運用する人間・AI
時間スケール  ：一つの運用プロジェクトのライフサイクル
空間スケール  ：単一用途から複合システムまで
評価関数      ：モジュール選択の操作的有用性
```

### 有効範囲

```text
・RDL 系の借用モジュールを選ぶ際の手順として
・「なぜそのモジュールを使うか」の根拠を明示するために
・解像度と計算負荷のトレードオフを操作するために
・ハイブリッド運用の境界 B を明示するために
```

### 推定破壊領域

```text
・目的が複数あり評価関数が競合する場合
    → どちらの目的に対して解像度を最適化するか決まらない

・計算資源が実行中に大きく変動する場合
    → 追従運用のコストが設計コストを超える

・借用モジュール自体が未整備の領域
    → 選べるモジュールがない場合は判定が機能しない

・目的そのものが運用中に大きく変わる場合
    → 引き直しが間に合わない
```

### 残存構造

```text
・「目的から入る」という順序の原則
・「できる」と「必要」を分離する発想
・せめぎあいを一回で解決しない追従運用の姿勢
・モジュール間境界 B の明示という原則
```

### 移植経路

```text
・AI システム設計における計算資源配分の判断論として
・シミュレーション設計全般における解像度設計論として
・RDL 以外の多層モジュール系の切替設計論として
```

### 判定

```text
RDL 系の運用層として有効。

ただし本文書は「正解の判定表」ではなく「手順論」である。
目的・制約・モジュール構成が変われば引き直す。

本文書も SILN であり、絶対化されてはならない。
破断条件が見えたら改訂する。
```
