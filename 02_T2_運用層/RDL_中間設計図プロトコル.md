# RDL中間設計図プロトコル
*T2：運用層 / OP / ACTIVE v0.1*
*依存：仮設という操作 / SILN / RDL_Core / 文書性の限界 / 翻訳接続系 / 構造破壊学*

---

## ■ 0. 一文定義

> RDL中間設計図プロトコルとは、自然言語の曖昧な意図をB・SILN・W_ij・M_B・ξ・禁止補完・破断条件へ分解し、後続の実装・創作・制度設計・研究・対話に渡せる構造として保持する手順である。

---

## ■ 1. なぜ「中間設計図」か

人間の中にある意図は、最初から構造になっていない。

```
意図の実態：
  曖昧・断片的・矛盾を含む・時間変化する・言語化されていない
```

この曖昧な意図を直接「実装」に渡すと、何が起きるか。

```
・AI、他者、未来の自分に意図が伝わらない
・実装者が意図を勝手に補完する（禁止補完の発生）
・「動いているが意図と違う」状態が後から発覚する
・修正のたびに意図から再確認しなければならない
```

中間設計図はこれを防ぐ層である。

```
曖昧な意図
  ↓  [中間設計図化]
B・SILN・W_ij・M_B・ξ・禁止補完・破断条件
  ↓  [実装・変換]
コード / 文書 / 制度 / 創作 / 行動
```

中間設計図は完成物ではない。意図をAIや他者が扱える構造に落とした**仮設**である。

---

## ■ 2. 中間設計図の構成要素

```
B        = 目的状態（何が達成された状態か。一文で表せる粒度）
SILN     = 安定させる構造パターン（モジュール・役割・概念・制度単位）
W_ij     = SILN間の接続・依存・データ流・関係の方向
M_B      = 固定されている前提・慣性（変えない・変えられない制約）
ξ       = 不確かな部分・変更可能性がある部分・未確定領域
禁止補完  = 実装者・AIが確認なしに決定してはいけない領域
破断条件  = この構造が機能しなくなる条件
```

これらを明示することで、意図が構造として渡せる状態になる。

---

## ■ 3. 共同分解ループ

中間設計図は一人で書くものではない。意図を持つ人とそれを構造化する側（AI・協働者）が対話しながら固める。

```
[共同分解ループ]

意図を持つ側：「〜がしたい」（自然言語・断片・直感でよい）
    ↓
構造化する側：B候補・SILN候補・ξ候補・禁止補完候補を提示
             「Bはこう理解しました。SILNはこの3つが候補です。
              ここはξとして未確定のままにしておきますか？」
    ↓
意図を持つ側：「BはOK。でもSILNのここは違う。これはM_Bにしてほしい」
    ↓
構造化する側：修正して再提示
    ↓
    ┌── まだズレがある → 上に戻る
    └── 固まった → 中間設計図として確定・後続へ渡す
```

終了条件：

```
□ BがAIとユーザーで一致している
□ 主要なSILNが列挙されている
□ ξと禁止補完が明示されている
□ W_ijの大まかな方向が見えている
□ 破断条件が少なくとも1つ記述されている
```

## ■ 4. 中間設計図テンプレート

```
B        = [達成したい状態]

SILN     = [安定させる構造の名前]
          └ 役割：[何をする単位か]
          └ 条件：[Bが成立する条件]

W_ij     = [SILN間の接続・依存・データ流]
          例：
          SILN_A → SILN_B → SILN_C

M_B      = [固定されている前提・変えない制約]

ξ       = [不確かな部分・変更可能性がある部分]

禁止補完  = [確認なしに決定してはいけない領域]

破断条件  = [この設計が機能しなくなる条件]
```

---

## ■ 5. 応用先

中間設計図は後続の実装・変換の種類を選ばない。

| 応用先 | 後続操作 | 固有の追加要素 |
|---|---|---|
| コード生成 | プログラム言語への変換 | 熱源可視化・言語選択・ベル的検証 |
| 文書作成 | 自然言語文書への展開 | 読み手のB・文書性の限界 |
| 制度設計 | 規則・組織・手順への変換 | スケール・時間変化・権力構造 |
| 創作 | 物語・世界観・キャラクターへの展開 | 感情・偶発性・ξの積極的活用 |
| 研究設計 | 仮説・実験・測定系への変換 | 反証可能性・観測誤差 |
| 会話整理 | 対話の構造化・合意形成 | 相手のM_B・価値観のξ |

各応用先には固有の追加要素が存在するが、共同分解・禁止補完・破断条件は全応用先に共通する。

---

## ■ 6. 禁止補完

実装者・AIが確認なしに決定してはいけない領域。

```
ξ    = 未確定領域
禁止補完 = 実装者・AIによる無断SILN化禁止領域
```

禁止補完をξとして明示しておくことで、「気を利かせた補完」による意図の暗黙化を防ぐ。未記載の場合、実装者・AIはξを自己判断で閉じる。

---

## ■ 7. 破断条件

このプロトコルは以下の場合に破断する。

```
1. Bが一文で表せないほど巨大なまま後続操作に進んだ場合
2. ξが明示されず、実装者・AIが未確定領域を勝手に補完した場合
3. W_ijが未記述で、依存関係が後続操作時に暗黙化した場合
4. M_Bが曖昧で、前提が後から揺れた場合
5. 誤差観測を後続操作の修正だけで処理し、中間設計図へ戻らなかった場合
```

特に5が重要。中間設計図を挟む意味は、**設計仮設の更新履歴を残すこと**にある。意図の変化が中間設計図を経由せず後続操作にのみ反映されると、次回の共同分解時に起点を失う。

---

## ■ 8. 仮設としての中間設計図

中間設計図は確定した仕様書ではない。

```
・引き直せる
・誤差を受け取れる
・壊れても再構成できる
・閉じない
```

「このBは正しい」ではなく「このBは今耐久している」という保持が正しい運用である。（→ 仮設という操作）

後続操作で誤差が出たとき、後続操作を修正するだけでなく、中間設計図のどの仮設がズレていたかを問い直し、共同分解ループに戻る。

---

## ■ 9. 一文圧縮

> 曖昧な意図をB・SILN・W_ij・M_B・ξ・禁止補完・破断条件へ共同分解し、後続のあらゆる実装・変換に渡せる仮設的構造として保持するプロトコル。

---

## ■ 10. 超圧縮

> Bを仮設し、SILNを切り、W_ijを見て、ξを残し、禁止補完を置き、誤差で戻る。
