# 圏論的構造破壊検査プロトコル
*T2：運用層 / OP / ACTIVE v0.1*
*依存：RDL_Core / SILN / 構造破壊学_前提文書 / RDL_圏論借用モジュール*

---

## ■ 0. 位置づけ

本プロトコルは、圏論の語彙を使ってRDL構造の破断点を探す操作手順である。

```
圏論借用モジュール（T3） = 語彙の接続
本プロトコル（T2）       = その語彙で破断点を探す操作
```

圏論で「綺麗に書けない場所」を探すことが目的であり、綺麗に書くことが目的ではない。

---

## ■ 1. 検査の基本構造

```
RDL構造を圏論的に図式化する
  ↓
可換性・合成可能性・関手保存性を問う
  ↓
成立する → その範囲は安定
不成立   → 破断候補として記録
  ↓
破断候補をRDL変数で分類する
（H / ξ / B設定ミス / W_ij破断 / 翻訳破断）
  ↓
B再仮設またはSILN再編へ
```

---

## ■ 1.1 検査前処理：圏論化の範囲設定

検査に入る前に以下を明示する。

```
対象化するSILN：
  何をObjectとして扱うか

射として扱うW_ij：
  どの関係・依存・変換をMorphismとして扱うか

等価判定B：
  どの差分まで「同じ」とみなすか（次節参照）

圏論化しないξ：
  図式外に残す未確定部分・違和感・禁止補完領域
```

この前処理を省くと、圏論図式が綺麗に見えても、B設定自体が不明なままになる。図の「綺麗さ」がB設定を隠蔽する。

---

## ■ 1.2 等価判定の前提：≈_B

本プロトコルにおける `=` は、数学的完全一致ではなく、
**観測者Bにおける操作的同値 `≈_B`** として扱う。

```
g ∘ f ≈_B h
```

とは、異なる経路を通った結果が、現在のBにおいて**同じSILNとして扱えること**を意味する。

```
完全一致しない → それだけでは破断ではない
現在のBで許容できない差分が出た → H蓄積・ξ残存・B設定ミスとして記録
```

≈_Bは固定ではなく、Bが変われば等価判定も変わる。AIコーディングでは特にこの点が重要で、「過剰に厳密に可換を要求する」と破断候補を見失う。

---

## ■ 2. 検査1：可換性テスト

### 概要

同じ出発点から異なる経路を辿ったとき、同じ結果に至るかを確認する。

```
A
├─ f → B ─ g → C
└─ h ────────→ C

g ∘ f = h なら可換（安定）
g ∘ f ≠ h なら非可換（破断候補）
```

### RDLへの読み方

```
可換   = 経路によらず同じSILN・同じ状態に至る
非可換 = 経路依存のH蓄積が発生している
```

### 破断候補の分類

| 非可換の原因 | RDL的解釈 |
|---|---|
| 操作順序で結果が変わる | H蓄積・整合慣性が経路に依存 |
| 同じBのはずが別SILNになる | B設定が実態と合っていない |
| 途中でξが介入している | ξ残存が経路差を生んでいる |
| W_ijの方向が逆になる | 依存関係の設定ミス |

### 実用例（AIコーディング）

```
要求A
├─ f → RDL設計B ─ g → コードC
└─ h → UML設計D ─ k → コードC'

g ∘ f ≈ k ∘ h にならない
→ 同じ要求から別の実装が生まれている
→ どちらかの経路でBが曲がった・禁止補完が入った
```

---

## ■ 3. 検査2：合成可能性テスト

### 概要

操作連鎖が定義できるかを確認する。

```
f: A → B
g: B → C
g ∘ f: A → C  // 合成可能

f: A → B
g: X → C  // B ≠ X → 合成不可
```

### RDLへの読み方

```
合成可能   = SILNの変換パイプラインが成立する
合成不可能 = 操作プロトコルの破断・接続ギャップが存在する
```

### 破断候補の分類

```
合成できない場所 = W_ijが未定義・インターフェースが不一致・
                   中間SILNが欠落している
```

---

## ■ 4. 検査3：関手保存テスト

### 概要

翻訳操作（関手）が構造を保っているかを確認する。

```
関手 F: C → D が構造を保つなら
  F(g ∘ f) = F(g) ∘ F(f)  // 合成が保たれる
  F(id_A) = id_{F(A)}      // 恒等が保たれる
```

### RDLへの読み方

```
関手保存 = 翻訳接続系が構造を正しく変換している
保存失敗 = 翻訳後にW_ij方向・合成可能性・B設定が崩れている
```

### 適用場面

```
RDL → コード変換  : コード生成後にW_ij方向が保たれているか
RDL → UML変換    : UML図がSILN間の依存を正しく反映しているか
外部理論 → RDL   : 借用モジュールが元の関係構造を保っているか
```

---

## ■ 5. 検査4：自然変換テスト

### 概要

二つの翻訳方式（関手）が整合しているかを確認する。

```
F, G: C → D の二つの関手に対して
自然変換 η: F ⟹ G が存在するなら整合
存在しないなら二つの翻訳方式が衝突している
```

### RDLへの読み方

```
自然変換あり = 二つの借用モジュール・翻訳方式が整合している

自然変換が構成できない / 自然性が破れる =
  二つの翻訳方式が、対象ごとには似て見えても、
  射・変換経路を含めると整合しない
```

「存在しない」の厳密証明より、**局所一致しているが変換経路で壊れる**場所を探すことが運用上の主な使い方。これはそのままH蓄積・ξ残存の発見地点になる。

### 適用場面

```
NAMとFluctMODを同じ対象に使う場合
UMLとRDL記述の両方を並用する場合
複数のAIモデルが同じ仕様を別解釈した場合
```

---

## ■ 6. 検査5：極限・余極限テスト

### 概要

複数のSILNを統合・貼り合わせる際に共通構造が存在するかを確認する。

```
極限   = 複数対象を同時に満たす整合点を探す操作
         集合的には「共通部分」に近い比喩で読める（操作的近似）

余極限 = 複数対象を矛盾なく貼り合わせる操作
         集合的には「共通包含」に近い比喩で読める（操作的近似）
```

### RDLへの読み方

```
極限が取れない  = 複数SILNの統合不能・共通M_Bが存在しない
余極限が壊れる = SILNを貼り合わせたときに境界Bが衝突する
```

### 適用場面

```
複数チームの設計を統合する場面
複数借用モジュールを同一系で使う場面
大きなB（システムB）に小さなBを貼り合わせる場面
```

## ■ 7. 検査結果の分類と次アクション

| 検査 | 不成立の意味 | 次アクション |
|---|---|---|
| 可換性テスト | 経路依存H・B設定ミス | B再仮設・W_ij修正 |
| 合成可能性テスト | 操作プロトコル破断 | 中間SILNの追加・接続修正 |
| 関手保存テスト | 翻訳接続破断 | 翻訳接続系の見直し |
| 自然変換テスト | 翻訳方式の衝突 | 借用モジュールの選択・整合 |
| 極限テスト | 統合不能 | B設計の見直し・統合断念 |
| 余極限テスト | 境界衝突 | B境界の引き直し |

どの検査でも、**不成立 = ξ残存・破断候補**として記録し、構造破壊学の破断条件リストに追加する。

---

## ■ 8. 圏論で消えないものの扱い

圏論の枠組みで記述できない部分はξとして扱う。

```
圏論で記述できた部分   → 構造として安定している領域
圏論で記述できない部分 → ξ残存・B再仮設候補
```

これはプロトコルの失敗ではない。**ξの可視化として積極的に使う**。

---

## ■ 9. 破断条件

このプロトコル自体が機能しなくなる条件：

```
1. 圏論の語彙をRDLのCoreとして採用しようとする
   → RDLのξ残存性・動態は圏論の記述外

2. 「可換にならない = 設計が悪い」と断定する
   → 非可換はξ残存の可能性もある。破断候補として扱い、断定しない

3. 検査をパスしたから安定と判断する
   → 圏論が照らせる範囲は限定的。検査外のξは常に残る

4. 全ての構造を圏論化しようとする
   → 過剰な形式化はB設定を硬直させH蓄積を増やす
```
