# RDFエージェント跳躍モジュール v0.1

---

## 0. 位置づけ

本稿は、関係力学言語（RDF）の枠組みに基づき、エージェント内部における**跳躍（Leap）**をどのような機構として実装するかを整理するための設計文書である。

ここでいう跳躍とは、単なるランダム変異でも、魔法のような創造でもない。

**未整合の蓄積によって生じた誤差熱が閾値を超えたとき、既存構造の再接続候補が複数生成され、その一部のみが試行を経て定着する現象**として扱う。

したがって本モジュールの焦点は、

- 跳躍候補の生成
- 候補の限定試行
- 候補の大半の散逸
- ごく一部の再現的固定

という一連の流れにある。

---

## 1. 基本認識

跳躍は「突然のひらめき」と見えることがある。
しかしRDF的には、それは**内部に蓄積した未整合が、ある素流圧をきっかけに再接続された結果**と解釈する。

つまりエージェントに必要なのは、

- 未整合を記録する仕組み
- 熱として蓄積する仕組み
- 既存構造を再接続する仕組み
- 生成候補を選別し固定する仕組み

である。

跳躍後の候補の多くは使えない。
むしろ**ほとんどは消えていく**。
生き残るのは、環境との整合、再現性、基層的必要性を満たした一部のみである。

---

## 2. モジュール全体像

本モジュールは、以下の5部位から構成される。

### 2.1 通常整合部

現在の整合慣性 `M_core` に基づき、入力を解釈し、予測し、行動を選択する。
通常時のエージェントはここで動く。

### 2.2 誤差熱蓄積部

入力・予測・結果のズレを未整合として記録し、熱 `H` として蓄積する。
ここでは単発の失敗をすぐに消さず、ある程度持続する違和感として保持する。

### 2.3 跳躍候補生成部

熱が閾値を超えたとき、既存構造を再接続し、複数の候補構造 `M_trial` を生成する。
これは創造器ではなく、**再接続器**である。

### 2.4 限定試行部

生成された候補を全面採用せず、局所状況・短期試行・内部シミュレーションによって試す。

### 2.5 定着選別部

候補の再現性、環境整合、生存寄与、反復可能性を評価し、通過したものだけを `M_core` に統合する。
通過しなかったものは散逸する。

## 3. 状態変数

最小構成として、エージェントは以下の内部状態を持つ。

```text
M_core       : 現在の整合慣性
H            : 誤差熱
θ            : 跳躍閾値
traits       : 基層パラメータ
memory       : 経験記録
trial_models : 仮説構造群
```

補助的には以下も有効である。

```text
E_pred        : 予測誤差
E_reward      : 報酬矛盾
E_social      : 他者との未整合
E_identity    : 自己構造との矛盾
E_novelty     : 未知対象による違和感
B             : 退屈由来の仮想熱
```

---

## 4. 熱源

誤差熱 `H` は単一の失敗ではなく、複数の未整合の合成として扱う。

### 4.1 主な熱源

- 予測が外れ続ける
- 行動しても期待した報酬が得られない
- 同じ対象が状況によって別の振る舞いを見せる
- 他者との相互作用が説明不能である
- 自己ラベルと行動結果が一致しない
- 未知対象に遭遇し解釈が安定しない

### 4.2 仮想熱

人間型・動物型エージェントを想定する場合、誤差だけでなく**退屈**も熱源としてよい。

退屈とは、誤差が少なすぎる環境において発生する探索促進用の仮想熱である。

したがってエージェントは、

- 苦しすぎても跳躍する
- 安定しすぎても探索へ向かう

という二方向の熱源を持ちうる。

---

## 5. 跳躍条件

熱が閾値を超えたとき、跳躍モードへ入る。

```text
if H > θ:
    enter leap mode
```

ただし完全に機械的に発火させる必要はない。
以下の要素で重み付けしてよい。

- 危険状況か
- 未知性が高いか
- 他者影響が大きいか
- 既存構造での失敗が反復しているか
- 現在の資源量に余裕があるか

つまり、**跳躍可能性**と**跳躍コスト**の両方を見る。

---

## 6. 跳躍候補生成

### 6.1 基本原理

跳躍はゼロからの生成ではなく、既存構造の再接続である。

よって候補生成では、

- 記憶にある類似事例
- 別用途で使われた行動系列
- 既知ラベルの境界変更
- 目標優先順位の入れ替え
- 自己／他者境界の再設定
- 新しい中間ラベルの仮設置

などを用いる。

### 6.2 候補生成の例

未知の植物を見つけた場合：

- 食料候補
- 毒候補
- 薬候補
- 道具候補
- 無視対象

敵対行動を受けた場合：

- 即逃走
- 反撃
- 威嚇のみ
- 仲間招集
- 相手の意図観測

新しい環境を見つけた場合：

- 危険地帯
- 資源地帯
- 巣候補
- 通過地点
- 監視対象

ここで重要なのは、**一つの正解をすぐ決めない**ことである。

---

## 7. 限定試行

生成された候補は、いきなり中核へ落とさない。
まずは仮採用し、局所的に試す。

### 7.1 試行方法

- 内部シミュレーション
- 低リスク場面での限定行動
- 反復可能な短期試験
- 他個体の観察との照合
- 部分的適用

### 7.2 仮採用層

エージェント内部は少なくとも以下の3層を持つとよい。

```text
M_core   : 通常運用層
M_trial  : 仮説試行層
leap_log : 跳躍候補の記録
```

この構造により、跳躍は全面的自己改変ではなく、
**試しに別の構造で動いてみる局所実験**として扱える。

---

## 8. 候補の散逸

本モジュールの重要点は、跳躍候補の大半が消えることである。

- 一度だけうまくいった
- 偶然の一致だった
- 高コストで続かない
- 生存には寄与しない
- 他状況へ拡張できない
- 内部的に説明不能

このような候補は `M_core` へ入れず、散逸させる。

ここで散逸とは、完全消去に限らない。

- 低優先度記憶へ移す
- 類似状況のみ参照可能にする
- 一時ラベルとして保留する
- 熱履歴だけ残す

などでもよい。

---

## 9. 定着条件

候補が定着するためには、少なくとも以下を満たす必要がある。

### 9.1 実装可能性

身体、資源、時間、環境制約の中で実行できるか。

### 9.2 基層的必要性

生存、報酬、安全、仲間維持などに寄与するか。

### 9.3 再現性

一回だけでなく、条件を変えても繰り返し成立するか。

### 9.4 圧縮可能性

内部ラベルとして扱えるか。記憶・判断・伝達に再利用できるか。

### 9.5 反復耐久性

続けて使っても破綻しにくいか。

この条件を通った候補のみが、新たな整合慣性 `M'` の一部として固定される。

---

## 10. D1〜D4の跳躍対応

本モジュールでは、基層パラメータのうちドーパミン系列を以下のように位置づける。

### D1：行動開始

未知や誤差に対して踏み出す力。
跳躍候補の試行開始に関わる。

### D2：抑制

不要候補、危険候補、過剰候補を止める。
暴走的跳躍を抑える。

### D3：再現性確認＋反復固定ループ

跳躍候補に対して反復試行を課し、
**再現するものだけを整合慣性として固定する選別機構**。

本モジュールにおいてD3は極めて重要である。
跳躍候補の死骸の山から、生存候補だけを選び出す役割を持つ。

### D4：退屈

誤差が少なすぎる状態において仮想熱を生み、探索を促す。
跳躍発火の前段にある探索圧として働く。

---

## 11. D3の特別な役割

D3は単なる習慣化ではない。
本モジュールでは、以下の機能を持つ。

1. 候補の反復試行要求
2. 偶然成功の排除
3. 条件差の観察
4. 再現パターンの抽出
5. 反復成立した候補の固定

したがって、D3は

> 跳躍で生成された候補構造に対して、反復運用を通じて再現性を検査し、生き残ったものを整合慣性として固定する機構

と定義できる。

### D3が強すぎる場合

- 固定が早すぎる
- 柔軟性が落ちる
- 局所解を中核化しやすい

### D3が弱すぎる場合

- 候補がいつまでも流動する
- 学習が定着しない
- 新構造が文化・習慣化しない

---

## 12. 個体差

基層パラメータの違いにより、跳躍の型は大きく変わる。

### 高D1・低D3

- 候補生成と試行は多い
- 固定が弱い
- 跳躍乱発型

### 低D1・高D3

- 新規試行は少ない
- 通ったものは強く固定
- 保守固定型

### 高D1・高D3

- 探索も固定も強い
- 発見と定着が両立しやすい
- 強力だが偏ると危険

### 低D2

- 危険候補まで試しやすい
- 跳躍暴走の危険

### 高D4

- 安定しすぎると探索へ向かいやすい
- 退屈耐性が低く、新規接触を増やす

---

## 13. 統合型と排出型

跳躍は必ずしも良いものではない。

### 統合型

誤差を自己構造の再編として処理する。
柔らかく構造が拡張される。

### 排出型

誤差を外部へ押し付け、自己構造を守る。
局所的には安定するが、適応力を失いやすい。

エージェント設計においては、排出型が起きる可能性も前提とする必要がある。
たとえば失敗が続いた際、

- 環境が悪い
- 他者が悪い
- 既存ラベルは疑わなくてよい

といった構造が形成されると、熱は下がったように見えても柔軟性が落ちる。

したがって、

- 単一原因への固定化を監視する
- 代替仮説を複数要求する
- 外部責任ラベルの増加時に自己検証を強制する

といった安全弁が望ましい。

## 14. 簡易アルゴリズム

```python
step(F):
    prediction = interpret(F, M_core)
    E = calc_error(F, prediction)
    H = H + error_heat(E) + boredom_heat() - dissipate()

    if H > theta:
        candidates = generate_candidates(M_core, memory, traits, F)

        for c in candidates:
            result = trial(c)
            score = evaluate(
                feasibility=result.feasible,
                survival=result.useful,
                reproducibility=result.repeatable,
                compressibility=result.labelable
            )

            if score > fix_threshold:
                M_core = integrate(M_core, c)
                H = release(H)
            else:
                scatter(c)
```

この擬似コードの本質は単純である。

**熱がたまる → 候補が出る → 少し試す → ほとんど捨てる → 一部だけ残す**

である。

## 15. 社会化への拡張

複数エージェント系では、個体内部の跳躍だけでなく、媒介構造が重要となる。

### 役割例

- 発見者：候補を生みやすい
- 翻訳者：候補を共有可能な形式へ変換する
- 検証者：再現性を調べる
- 保守者：固定後の安定運用を担う
- 拡散者：群れへ広げる

この構造により、個体の跳躍が社会的整合慣性へ変換される。

## 16. まとめ

RDFエージェントにおける跳躍とは、
**誤差熱の限界突破によって大量の新構造候補が生まれ、その大半が消え、ごく一部だけが再現性検査を通過して整合慣性として固定される現象**である。

ここで核心となるのは以下である。

- 跳躍は創造ではなく再接続である
- 候補は複数生成される
- 候補の大半は散逸する
- D3が再現性検査と固定化を担う
- 定着は反復と環境整合によってのみ起こる

圧縮して言えば、

> 跳躍モジュールとは、未整合を熱として蓄積し、臨界到達時に既存構造の再接続候補を生成し、そのうち反復的に成立するものだけを新たな整合慣性として定着させる選別機構である。
