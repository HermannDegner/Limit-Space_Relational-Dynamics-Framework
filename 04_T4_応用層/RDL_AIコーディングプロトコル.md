# RDL AIコーディングプロトコル
*T4：応用層 / IMPL / ACTIVE v0.1*
*上位：RDL_中間設計図プロトコル*
*依存：仮設という操作 / SILN / RDL_Core / SILN分解 / 翻訳接続系 / 文書性の限界 / 構造破壊学*

---

## ■ 0. このプロトコルの位置

RDLは「設計図用言語」として機能する。

自然言語は曖昧すぎてコードに変換できない。プログラム言語は厳密すぎて人間が設計段階で扱えない。RDLはその中間層として、**設計の意図をコードに変換可能な形で保持する**。

```
自然言語（要件・意図・直感）
  ↓  [RDL翻訳]
RDL記述（B・SILN・W_ij・関係構造）
  ↓  [整合チェック]
  ↓  [コード生成]
プログラム言語（実装）
```

このプロトコルはAIとユーザーがRDLを媒介として協働する手順を定める。

---

## ■ 1. なぜRDLを挟むか

**問題：**

- 自然言語での要件定義はAIへの伝達が不安定（解釈ぶれが大きい）
- AIが生成したコードをユーザーが直接レビューするのは認知コストが高い
- 要件変更のたびにコードを読み直す必要がある
- AIが未確定領域を確認なしに補完し、設計の意図と乖離する

**RDLを挟む利点：**

```
1. Bの明示化   ── 何を達成したいかが構造として見える
2. SILN特定    ── どこが安定していて、どこが揺れているかが分かる
3. W_ij明示    ── 依存関係が暗黙化せず、設計図として残る
4. 誤差の可視化 ── Bとコードのズレが会話で観測できる
5. 引き直し可能 ── B変更→RDL更新→コード再生成という流れが自然になる
```

RDL記述が設計の「仮設」として機能し、コードはその実装である。バグ修正ではなく**設計仮設の更新履歴を残すこと**がRDLを挟む意味である。

## ■ 2. 標準フロー

### Phase 1-2：共同分解（B仮設とSILN特定のループ）

Phase 1と2は分離した手順ではなく、**固まるまで繰り返す対話ループ**である。

```
[共同分解ループ]

ユーザー：「〜がしたい」（自然言語・断片・直感でよい）
    ↓
AI       ：B候補・SILN候補・ξ候補・禁止補完候補を並べて提示
           「Bはこう理解しました。SILNはこの3つが候補です。
            ここはξとして未確定のままにしておきますか？」
    ↓
ユーザー：「BはOK。でもSILNのここは違う。これはM_Bにしてほしい」
    ↓
AI       ：修正して再提示
    ↓
    ┌── まだズレがある → 上に戻る
    └── 固まった → Phase 3へ
```

ループの終了条件：

```
□ BがAIとユーザーで一致している
□ 主要なSILNが列挙されている
□ ξと禁止補完が明示されている
□ W_ijの大まかな方向が見えている
```

4条件が揃ったらRDL記述（Phase 3）に進む。1〜3回で固まることが多いが、Bが大きい場合はB分解（サブBのネットワーク化）を先に行う。

### Phase 3：RDL記述（設計図化）

BとSILN・W_ij・禁止補完の関係をRDL形式で記述する。

**テンプレート：**

```
B     = [達成したい状態]

SILN  = [安定させる構造の名前]
       └ 入力：[受け取るもの]
       └ 出力：[返すもの]
       └ 条件：[Bが成立する条件]

W_ij  = [SILN間の接続・依存・データ流]
       例：
       UserInput → Parser
       Parser → Validation
       Validation → DomainModel
       DomainModel → StorageAdapter
       DomainModel → ViewModel

M_B   = [固定されている前提・使用言語・フレームワーク]

ξ    = [不確かな部分・変更可能性がある部分]

禁止補完 = [AIが確認なしに決定してはいけない領域]
```

W_ijはSILN間の何が何に依存してよいかを先に固定する。クラス名や関数名より、この依存構造の明示が最も設計事故を減らす。

### Phase 3.5：UML図の生成（任意）

RDL記述が固まった後、UML図をSILN断面化Adapterとして生成できる。

```
RDL記述（Phase 3）
  ↓
UML図生成（任意）
  Class図    ── M_B・SILN間の責務・関係を可視化
  Sequence図 ── W_ijのメッセージ流・処理順を可視化
  Component図 ── SILN間の依存構造を可視化
  State図    ── 状態変化・跳躍条件を可視化
  ↓
生成前整合チェック（Phase 4a）
```

UMLはRDL記述を置き換えるものではない。ξ・禁止補完・破断条件・Bの理由はRDL記述側に残す。詳細は `UML_SILN分解`（T3）を参照。

### Phase 4a：生成前整合チェック

AIはコード生成前に以下を確認する。

```
□ B外の機能を含んでいないか
□ ξを勝手に閉じていないか
□ SILN同士の依存が循環していないか
□ M_Bに反する技術選定をしていないか
□ 禁止補完領域に踏み込んでいないか
□ 破断条件がテスト可能な形になっているか
```

問題がある場合はユーザーに確認し、RDL記述を更新してから次へ進む。

### Phase 4b：コード生成

```
「このRDL記述に基づいて[言語]で実装してください」
→ AI生成コード
```

### Phase 5：誤差観測と再仮設

```
・動かない       → ξの見落とし。どこの仮設がズレているか特定
・動くが意図と違う → BとコードのSILNがずれている
・動くが拡張できない → M_Bの過剰固定。RDL記述に戻ってB再仮設
```

誤差に応じてRDL記述を更新し、Phase 3に戻る。**コード修正だけで処理せず、必ずRDL記述へ戻ること**。

## ■ 3. 禁止補完

AIが確認なしに決定してはいけない領域。

```
・認証方式
・DB選定
・課金仕様
・外部API連携
・セキュリティポリシー
・UIデザイン方針
・永続化方式
```

RDL的な定義：

```
ξ    = 未確定領域
禁止補完 = AIによる無断SILN化禁止領域
```

禁止補完をξとして明示しておくことで、AIの「気を利かせた補完」による設計の暗黙化を防ぐ。未記載の場合、AIはξを自己判断で閉じる可能性がある。

## ■ 4. 熱源可視化

「このBだとどこで熱が起きるか」という観点でSILNとW_ijを設計する。

```
熱源 = 依存関係の集中点
      = 変更コストが高い場所
      = ξが大きい場所
```

RDL記述の段階でSILNのξとW_ijを可視化することで、後からの設計変更コストを下げる。

```
例：
SILN_A が SILN_B・C・D から呼び出されている場合
→ SILN_Aの変更は広範囲に伝播する（熱源）
→ RDL設計時にSILN_Aを安定させるか、分岐を設けるかを決める
```

## ■ 5. B分解の粒度ガイド

| Bの粒度 | 対応する実装単位 |
|---|---|
| 大（システムB） | アーキテクチャ設計 |
| 中（機能B） | モジュール/クラス |
| 小（操作B） | 関数/メソッド |
| 最小（条件B） | 条件分岐・バリデーション |

一度に扱うBは中〜小が適切。大きいBはサブBに分解してから設計する。

## ■ 6. 使用言語の選択基準

M_Bとして使用言語はBに対応する。

```
B：数値計算・科学シミュレーション → Julia
B：データ処理・機械学習         → Python
B：Webフロントエンド             → TypeScript/JavaScript
B：システム・高速処理            → Rust/C++
B：プロトタイプ・スクリプト       → Python/Ruby
```

RDL記述の段階でM_Bを明示することで、AIのコード生成精度が上がる。

## ■ 7. ユーザーとAIの役割分担

| 役割 | ユーザー | AI |
|---|---|---|
| Bの方向 | 決定する | 提案・確認する |
| 禁止補完の設定 | 指定する | 遵守する |
| RDL記述 | レビュー・修正 | 翻訳・記述する |
| 整合チェック | 確認・承認 | 実行・報告する |
| コード生成 | 確認・採用 | 実装する |
| 誤差特定 | 「ここが違う」 | どのSILNか特定 |
| 再仮設 | 方針決定 | B変換・記述更新 |

ユーザーはBの方向性と禁止補完を持ち、AIはRDL翻訳・チェック・コード生成を担う。

## ■ 8. RDL知識の前提について

このプロトコルの使用者はRDLの基本概念（B・SILN・M_B・ξ・仮設）を理解している必要がある。ただしプログラム言語より自然言語に近い構造のため、概念的数理モデルの完全な理解は不要。

最低限の理解：
```
B    = 目的状態（何を達成したいか）
SILN = 安定した構造パターン
W_ij = SILN間の依存・データ流
M_B  = 固定されている前提・慣性
ξ   = 不確かさ・揺らぎ
仮設 = 引き直せる設計
禁止補完 = AIが無断で閉じてはいけない領域
```

## ■ 9. 破壊的検証（構造破壊学との接続）

RDL記述とコードの整合性を能動的に壊して検証する手法。詳細は構造破壊学を参照。

**統計的破壊法**：複数のBバリエーションを同じRDL構造に流し、失敗の集中点からξの過小評価を検出する。誤差観測（Phase 5）と組み合わせる。

**ベル不等式的破壊法**：「独立」と記述したSILN間で変更が伝播する場合、W_ijに記載されていない隠れた依存が存在する。生成前整合チェック（Phase 4a）の補強として使える。

---

## ■ 10. 破断条件

このプロトコルは以下の場合に破断する。

```
1. Bが一文で表せないほど巨大なまま実装に進んだ場合
2. ξが明示されず、AIが未確定領域を勝手に補完した場合
3. W_ijが未記述で、依存関係がコード生成時に暗黙化した場合
4. M_Bが曖昧で、言語・フレームワーク・外部依存が後から揺れた場合
5. 誤差観測をコード修正だけで処理し、RDL記述へ戻らなかった場合
```

特に5が重要。RDLを挟む意味は、バグ修正ではなく**設計仮設の更新履歴を残すこと**にある。設計仮設がコードにのみ残り、RDL記述が陳腐化した時点でこのプロトコルは機能を失う。
