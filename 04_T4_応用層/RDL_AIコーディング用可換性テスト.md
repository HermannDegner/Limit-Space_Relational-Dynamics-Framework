# RDL_AIコーディング用可換性テスト
*T4：応用層 / IMPL / ACTIVE v0.1*
*上位：RDL_AIコーディングプロトコル*
*依存：RDL_Core / SILN / 圏論的構造破壊検査プロトコル / RDL_圏論借用モジュール / 構造破壊学_前提文書*

---

## ■ 0. 位置づけ

本プロトコルは、AIコーディング文脈での可換性テストを実用手順として定める。

```
圏論的構造破壊検査プロトコル（T2） = 汎用の破断点探索手順
本プロトコル（T4）               = AIコーディング特化の可換性テスト
```

テスト対象は以下の経路差分：

```
要求A
├─ 経路1 ─ RDL設計 ─ コード
└─ 経路2 ─ UML設計 ─ コード

同じ要求から同じ結果に至るか？
→ 至らないなら、どこで何が曲がったか？
```

---

## ■ 1. テストの基本構造

```
同一の要求Aから複数の変換経路でコードを生成する
  ↓
経路間でコードを比較する
  ↓
差分あり → 非可換（破断候補）
差分なし → 可換（その範囲では安定）
  ↓
非可換の場合、差分の原因を特定する
```

---

## ■ 2. 標準の経路設定

### 経路1：RDL経由

```
要求A
  ↓ [共同分解：B・SILN・W_ij・ξ・禁止補完を明示]
RDL記述
  ↓ [生成前整合チェック]
コード C1
```

### 経路2：UML経由

```
要求A
  ↓ [UML設計：Class図・Sequence図等を生成]
UML設計図
  ↓ [コード生成]
コード C2
```

### 可換条件

```
C1 ≈ C2 なら可換（安定）
C1 ≠ C2 なら非可換（破断候補）
```

※完全一致ではなく「同一のB・SILN・W_ij構造を実現しているか」が判定基準。

---

## ■ 3. 非可換の分類

| 差分の種類 | RDL的解釈 | 次アクション |
|---|---|---|
| 依存方向が逆 | W_ij設定ミス | RDL記述に戻りW_ij修正 |
| クラス構成が別 | B設定が経路で曲がった | 共同分解ループに戻る |
| 未定義機能が増えた | 禁止補完が入った | どの経路で補完されたか特定・RDL禁止補完に追記 |
| 同一クラスが別名で重複 | 恒等射の不一致（B安定性崩壊） | SILN境界B再仮設 |
| 動作するが意図と違う | ξの見落とし | ξを再特定してRDL記述更新 |

---

## ■ 4. テスト手順

### Step 1：要求をRDL記述に固める（経路1起点）

```
共同分解ループを回し、以下を明示する：
  B        = 達成したい状態
  SILN     = 安定させる構造の単位
  W_ij     = SILN間の依存・データ流
  ξ       = 未確定部分
  禁止補完 = AIが確認なしに決定してはいけない領域
```

### Step 2：RDL記述からコードを生成（C1）

```
生成前整合チェック（Phase 4a）を通してからコード生成。
```

### Step 3：同じ要求からUML設計を生成（経路2起点）

```
RDL記述を参照せずに（または後で参照比較用に）
同じ要求AをUML図（Class図・Sequence図）に変換する。
```

### Step 4：UML図からコードを生成（C2）

```
UML図を入力としてコードを生成する。
```

### Step 5：C1とC2を比較する

```
比較観点：
  □ 同じクラス・関数が定義されているか
  □ 依存方向（W_ij）が一致しているか
  □ 禁止補完が入っていないか
  □ SILNの粒度が揃っているか
  □ ξを同じ場所で保持しているか（勝手に閉じていないか）
```

### Step 6：差分を記録・分類する

```
差分 → ■3の分類表で原因特定
原因 → RDL記述・禁止補完リスト・W_ij・SILN境界のいずれかを更新
```

---

## ■ 5. 可換性テストの適用タイミング

```
・要求が大きく変わったとき（B再仮設後）
・複数のAIが同じ仕様を別実装したとき
・設計レビューでRDL記述とコードのズレが疑われるとき
・UML設計とRDL設計を両方持っているプロジェクト
```

---

## ■ 6. 軽量版（クイックチェック）

フルテストが重い場合の簡易チェック：

```
1. RDL記述のW_ijを書き出す（SILN間の依存一覧）
2. 生成されたコード（またはUML図）の依存方向を抽出する
3. 二つを並べて方向の一致を確認する

差分があれば → 経路のどこかでW_ijが曲がった
差分がなければ → W_ij層では可換
```

この確認だけでも主要な破断候補を8割程度カバーできる。

---

## ■ 7. ξの可視化としての非可換

非可換は「設計ミス」ではなく、**ξの可視化**として扱う。

```
C1とC2が違う
= どちらかの経路でξが勝手に閉じられた
= ξを明示していなかった箇所が現れた
```

テストに失敗した場所は「修正すべき場所」ではなく「RDL記述に戻って再仮設する場所」として記録する。

---

## ■ 8. 破断条件

このテストが機能しなくなる条件：

```
1. 要求AがBとして明示されていない（何をテストしているかが不明）
2. RDL記述がコード生成後に更新されていない（経路1の起点が陳腐化）
3. 「可換にする」ことを目的化する
   → 可換化のためにξを潰すと破断候補が見えなくなる
4. UML図が要求Aではなくコードから逆生成された
   → 経路2がすでにC1に汚染されており比較にならない
```

---

## ■ 9. 一文圧縮

> 同一要求から複数経路でコードを生成し、差分を可換性テストで探し、非可換な場所をH蓄積・ξ残存・禁止補完侵入・W_ij破断として分類し、RDL記述へ戻ってB再仮設するプロトコル。

---

## ■ 10. 超圧縮

> 経路が違っても同じコードに着くか。着かない場所が破断候補。
