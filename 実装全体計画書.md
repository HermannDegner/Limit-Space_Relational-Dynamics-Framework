# RDF 汎用エージェントエンジン 実装全体計画書（MVP）

## 1. 目的

本計画書は、RDF（Relational Dynamics Framework）に基づく汎用エージェントエンジンを、
**段階的に実装・検証するための実行計画**を定義する。

中核方針は次の3点である。

1. レイヤー分離（Sensor / Neuro / Network / Base / Action）
2. プラグイン差し替え可能な入出力
3. 状態 `S`・熱 `H`・整合慣性 `M`・揺らぎ `ξ`・跳躍（Jump）を軸にした統一更新

---

## 2. スコープ

### 2.1 MVPスコープ（本計画で実装）

- 単一エージェントの `sense -> neuro -> base -> act` ループ
- 熱蓄積・散逸と跳躍トリガー
- 合成勾配ベース移動
- 跳躍候補生成と選別の最小実装
- 学習を「整合慣性更新」として実装

### 2.2 MVP外（後続）

- 高度な社会ネットワーク相互作用
- 大規模分散シミュレーション
- ドメイン特化プラグインの大量追加

---

## 3. 目標アーキテクチャ

```text
Sensor Layer  -> 未解釈作用束を勾配束へ変換
Neuro Layer   -> 勾配の変形・再重み付け・探索熱生成
Network Layer -> 個体間勾配の結合（MVPでは任意）
Base Layer    -> S/H/M/ξ/Jump を積分
Action Layer  -> 内部状態を外部行動へ変換
```

---

## 4. フェーズ別計画

## Phase 0: 仕様固定（0.5日）

### タスク

- 状態変数・主要パラメータ確定（`S`, `H`, `M`, `ξ`, `θ`）
- 各レイヤーの責務境界確定
- 最小I/F仕様（SensorPlugin / ActionPlugin）確定

### 成果物

- `docs/architecture.md`
- `engine/core/types.py`
- `engine/core/state.py`

### 完了条件（DoD）

- 状態遷移図・データフロー図が記述されている
- レイヤー間依存が一方向に整理されている

---

## Phase 1: Base Model 単体（1.5日）

### タスク

- `dS/dt`, `dH/dt`, Jump条件を実装
- 積分器（まずEuler）を実装
- NAM切替フック（低熱・低揺らぎ時）を用意

### 成果物

- `engine/core/equations.py`
- `engine/core/integrator.py`
- `engine/core/mode_switch.py`
- `tests/core/test_equations.py`

### DoD

- 一定入力で安定収束を確認
- 閾値超過でJumpフラグが立つ

---

## Phase 2: Sensor Layer（1日）

### タスク

- SensorPlugin抽象I/F実装
- 最小センサー1種（Text または Grid）実装
- センサー統合（複数入力の束化）

### 成果物

- `engine/plugins/sensor/base.py`
- `engine/plugins/sensor/text_sensor.py`（または `grid_sensor.py`）
- `engine/layers/sensor_layer.py`
- `tests/plugins/test_sensor_plugins.py`

### DoD

- センサー出力がBase入力形式に正規化される

---

## Phase 3: Action Layer（1日）

### タスク

- ActionPlugin抽象I/F実装
- 最小アクション1種（Move または API）実装

### 成果物

- `engine/plugins/action/base.py`
- `engine/plugins/action/move_action.py`
- `engine/layers/action_layer.py`
- `tests/plugins/test_action_plugins.py`

- 合成勾配で移動更新（点目標追跡から脱却）

### 成果物

- `engine/modules/movement/local_eval.py`
- `engine/modules/movement/flow_update.py`

### DoD

## Phase 5: 跳躍モジュール（2日）

### タスク

- 熱閾値到達時に候補構造を生成

- `engine/modules/leap/selection.py`

## 6. テスト計画

### 単体

- 数式更新の妥当性（符号・安定性）
- Jumpトリガー境界の再現
- Plugin I/F整合

### 結合

- 単一エージェントE2Eループ
- 移動 + 跳躍 + 学習の同時動作

### シナリオ

- 低熱安定シナリオ
- 高熱遷移シナリオ
- 退屈熱探索シナリオ

---

## 7. ログ・可観測性

最低限ログ:

- `H(t)` の推移
- Jump発火時刻
- 候補数 / 採択率
- `M` 更新量
- センサー別寄与重み

MVP時点ではCSV/JSONで十分。

---

## 8. リスクと緩和策

1. **複雑化リスク**
   - 緩和: PhaseごとにDoD固定、機能凍結して次へ進む
2. **過学習/硬直化**
   - 緩和: 退屈熱・揺らぎの下限を確保
3. **跳躍暴走**
   - 緩和: 候補評価の上限回数、採択ガード、ロールバック
