# UML_SILN分解
*T3：外部接続層 / BORROW / ACTIVE v0.1*
*依存：RDL_Core / SILN / RDL_中間設計図プロトコル / RDL_AIコーディングプロトコル*

---

## ■ 0. 一文定義

> **UMLとは、ソフトウェア設計対象を、複数の図式Bによって局所的にSILN化するための外部記法である。**

RDL的には、UMLは「設計そのもの」ではなく、
**設計対象の一部を安定断面として切り出すためのB群**。

RDLでは構造を「関係の中で一時的に安定している構造＝SILN」として扱い、SILN分解は「境界Bを仮定し、整合慣性M_Bを仮設する操作」とされている。

---

## ■ 1. 対象SILN：UML

```
SILN_UML =
  ソフトウェア設計対象を
  図式・ラベル・関係・制約によって
  実装前に一時安定化する記述構造
```

UMLが安定させようとしているものは、コードそのものではない。

安定化対象は、

```
要求
責務
構造
依存
状態
振る舞い
処理順
配置
```

のような、コード化前の設計関係。

---

## ■ 2. B：境界

UMLの本質は、**複数のBを持つこと**。

| UML図 | Bとしての意味 | 切り出すSILN |
|---|---|---|
| Use Case図 | 利用者・目的境界 | 何を提供するか |
| Class図 | 責務・型・関係境界 | 静的構造 |
| Object図 | ある時点の実体境界 | インスタンス状態 |
| Sequence図 | 時系列相互作用境界 | メッセージ流 |
| Communication図 | 相互接続境界 | 関係網 |
| State Machine図 | 状態遷移境界 | 状態変化 |
| Activity図 | 手順・分岐境界 | 処理流 |
| Component図 | 部品・依存境界 | モジュール構造 |
| Deployment図 | 実行環境境界 | 配置・通信 |

つまりUMLは単一の記法ではなく、

```
UML = {B_usecase, B_class, B_sequence, B_state, B_component, ...}
```

という**B集合**。

ここがAIコーディングでは重要。AIに自然言語だけ渡すとBが曖昧なので勝手に補完する。UMLを挟むと、どのBで安定化した構造なのかを明示できる。

---

## ■ 3. M_B：整合慣性

UMLのM_Bは、

```
「この図の範囲では、この設計構造として読める」
```

という慣性。

Class図なら：

```
M_B_class =
  クラス名 / 属性 / メソッド / 継承
  集約 / 関連 / 多重度 / 責務
```

Sequence図なら：

```
M_B_sequence =
  Actor / Object / Message / Return
  Alternative / Loop / Exception
```

---

## ■ 4. W_ij：関係重み

UMLの線・矢印・多重度・依存関係はRDL的にはW_ij。

```
ClassA -- ClassB
ComponentA -> ComponentB
User -> System
ObjectA -> ObjectB : message()
StateA -> StateB : event / guard / action
```

これらは単なる線ではなく、

```
どちらが依存するか
どちらが呼ぶか
どちらが所有するか
どちらが状態を変えるか
どちらが責務を持つか
```

を表す関係ラベル。

つまりUMLの図は、**ラベルM_Bネットワークを視覚化したもの**と読める。

---

## ■ 5. F：作用解釈

UMLにおけるFは、図を見た設計者・AI・実装者が受け取る作用。

```
F_human = 「このクラスが中心だな」
F_AI    = 「この責務でコードを生成すればよい」
F_impl  = 「この依存方向でファイル分割する」
```

同じUMLでも読む主体によってFが変わる。UMLは「図に描いたら意味が一意に伝わる」と思われがちだが、実際には違う。

```
UML図 ≠ 完全指定
UML図 = Fをかなり制限する設計断面
```

---

## ■ 6. ξ：残存揺らぎ

UMLが苦手なのはξの保持。UMLに残りやすいξ：

```
なぜその設計にしたのか
どこまで変えてよいのか
どの要求を優先したのか
失敗時に何を守るのか
非機能要件
禁止補完
未決定事項
設計者の違和感
将来の拡張余地
```

特にAIコーディングでは、Class図だけ渡すとAIは「足りない仕様」を自然に埋める。その補完が設計意図と一致するとは限らない。

UMLを使う場合でも、RDL側に必ず以下を残すべき：

```
ξ        ：未確定部分
禁止補完  ：AIが勝手に決めてはいけない部分
破断条件  ：この設計が壊れる条件
Bの理由   ：なぜこの図式で切ったか
```

---

## ■ 7. H：熱・誤差蓄積

UML運用で熱が溜まる場所：

```
図とコードが乖離する
図が古くなる
Class図だけ肥大化する
Sequence図が実装実態と合わなくなる
状態遷移図が例外処理を吸収できない
図が綺麗すぎて現場の泥を表せない
UMLを書くこと自体が目的化する
```

RDL的には `M_B_UML と M_B_code のズレ` がHとして蓄積している状態。

破断すると「UMLは役に立たない」「図を見るよりコードを読んだ方が早い」になる。

---

## ■ 8. M_Δ：再編相

UMLが壊れる局面は、設計再編の入口になる。

```
Class図が肥大化する
  → 責務境界Bが間違っている可能性

Sequence図が複雑化する
  → 同期/非同期、イベント駆動、責務分配が崩れている可能性

State図が爆発する
  → 状態モデルではなくポリシー/ルールエンジン化すべき可能性

Component図で依存が循環する
  → 境界設計が破断している可能性
```

つまりUMLの失敗は、**SILN分解の破壊検査ログ**として使える。

---

## ■ 9. SILN分解まとめ

```
対象：
  UML

B：
  図式ごとの境界設定
  UseCase / Class / Sequence / State / Component / Deployment ...

M_B：
  図の範囲で安定化された設計構造

W_ij：
  関連・依存・継承・集約・メッセージ・遷移・配置

F：
  設計者・実装者・AIが図から受け取る作用解釈

ξ：
  未決定事項、意図、非機能要件、禁止補完、図に落ちない判断

H：
  図と実装、図と要求、図同士の不整合

M_Δ：
  図の破綻をきっかけにした境界再設定

破断条件：
  UMLが実装より遅れる
  図が仕様書として固定化される
  動態・例外・意図を表せない
  AIが図外情報を勝手に補完する

残存構造：
  図式ごとのB設定能力
  関係可視化能力
  AIへの拘束力
  設計レビュー用の共通足場
```

---

## ■ 10. RDLとの接続

```
RDLなしのUML
  → 綺麗な図だが、意図・未確定性・破断条件が落ちる

UMLなしのRDL
  → 意図は残るが、コード生成時の構造拘束が弱い

RDL + UML
  → 意図と境界を保持しつつ、実装可能な断面を作れる
```

RDL中間設計図プロトコルにUMLを接続する場合、UMLはPhase 3（RDL記述）の後に生成するSILN断面として位置づける。

```
共同分解（Phase 1-2）
  ↓ B・SILN・W_ij・ξ・禁止補完が固まる
RDL記述（Phase 3）
  ↓ UML図を生成（RDL記述の視覚化・拘束層として）
生成前整合チェック（Phase 4a）
  ↓
コード生成（Phase 4b）
```

---

## ■ 11. 一文圧縮

> **UMLは、RDL中間設計図からコード生成へ渡すための、SILN断面化Adapterである。**

---

*v0.1：初版。UMLをRDL的にSILN分解。B（図式群）・M_B（図内整合慣性）・W_ij（矢印・依存）・F（作用解釈）・ξ（図に落ちない意図・未確定）・H（図とコードの乖離）・M_Δ（破断→再編）の全変数で記述。RDL中間設計図プロトコルへの接続としてPhase 3後のSILN断面化Adapterとして位置づけ。*
